# 결제 리팩터링 작업 계획서

> **문서 목적**: 결제(Payment)·환불(Refund)·위약금(Penalty) DB 구조 개편 작업의 전체 맥락을 기록합니다.
> 채팅 컨텍스트가 압축되더라도 이 문서를 참조하면 작업을 이어갈 수 있습니다.
>
> **최종 업데이트**: 2026-03-31
> **관련 PR**: (Phase별 PR 생성 시 업데이트)

## 1. 작업 배경 및 목적

### 1-1. 기존 DB 구조의 문제점

우유펫 모바일 앱에서는 예약 취소 시 **위약금을 PG사를 통해 별도 결제**하고, 기존 돌봄비 결제 건은 **전액 환불**하는 방식으로 이미 구현이 완료되어 있습니다. 그러나 관리자 백오피스 DB에는 이 로직이 아직 반영되지 않았습니다.

| 항목 | 기존 DB | 실제 서비스 로직 |
|------|---------|----------------|
| 위약금 결제 | `refunds` 테이블의 속성으로만 기록 (`penalty_amount`, `penalty_tx_id` 등) | PG사를 통한 독립적인 결제 건 (payments에 별도 행) |
| payments 테이블 | 돌봄비 결제만 존재 | 돌봄비 + 위약금 두 종류 존재 |
| 결제 유형 구분 | 없음 | `payment_type` 컬럼으로 구분 필요 |
| 환불 → 위약금 연결 | 없음 (refunds에 금액만 기록) | `penalty_payment_id` FK로 payments.id 참조 |

### 1-2. 작업 목적

1. **DB 스키마를 실제 서비스 로직에 맞게 정렬** — 위약금 결제를 payments 테이블에 독립 건으로 저장
2. **관리자 UI에서 돌봄비와 위약금을 명확히 구분하여 표시** — 결제내역 탭(돌봄만), 환불/위약금 탭
3. **향후 서비스 확장에 대비한 구조 마련** — payment_type으로 '훈련', '구독' 등 추가 가능

## 2. 확정된 비즈니스 로직

### 2-1. 환불 규칙 (취소 시점별 위약금 비율)

#### 위약금 발생 전제 조건

> **위약금은 유치원이 돌봄 요청을 수락하여 "예약확정"이 된 후에만 적용됩니다.**
> 수락대기 상태에서 보호자가 취소하면 **위약금 없이 전액 환불**됩니다.
> 이는 예약이 확정되기 전에는 실질적인 계약이 성립하지 않았기 때문입니다.
>
> 단, 위약금 산정 시 기준 시간은 **예약확정 시점이 아니라 결제 시점으로 소급 적용**됩니다.
> (예: 결제 후 48시간 뒤에 유치원이 수락하고, 수락 직후 보호자가 취소하더라도
> 결제 시점 기준으로 남은 시간을 계산하여 위약금 비율이 결정됩니다.)

| 규칙 | 취소 시점 | 환불 비율 | 위약금 비율 | 비고 |
|------|----------|----------|------------|------|
| 규정 1 | 등원 72시간 이상 전 | 100% | 0% | 전액 환불, 위약금 없음 |
| 규정 2 | 등원 24~72시간 전 | 50% | 50% | 위약금 별도 결제 발생 |
| 규정 3 | 등원 24시간 미만 전 | 0% | 100% | **취소 자체 불가 (환불 불가)** |
| 규정 4 (예외) | 결제 후 1시간 이내 | 100% | 0% | 등원까지 24시간 이하이면 적용 불가 |

> **핵심**: 위약금 > 0이면 보호자가 위약금을 PG사를 통해 **별도 결제**하고, 기존 돌봄비 결제 건은 **전액 취소(환불)** 처리됩니다.

### 2-2. 노쇼(No-Show) 처리 로직

**노쇼 시에는 위약금 별도 결제가 발생하지 않습니다.**

- 등원 24시간 이내에는 예약 취소 자체가 불가(환불 불가)하므로, 노쇼가 발생하면:
  - 기존 돌봄비가 **별도의 돌봄 서비스 수행 없이 그대로 유치원에 정산**됩니다
  - 보호자에게는 **이용정지 패널티만 적용** (1회: 7일 제한, 2회: 1개월 제한, 3회: 영구 정지)
  - 유치원에게는 **경고 및 신선도 하락** (1회: 경고, 2회: 영구 정지)
- **즉, 노쇼 건은 환불도 없고 위약금 별도 결제도 없습니다**
- payments의 상태는 `'결제완료'` 유지, refunds 건 생성 없음, settlements는 정상 정산 진행

### 2-3. 위약금 결제 흐름 (취소 시)

```
보호자 취소 요청
  → 등원까지 남은 시간 산정
  → 환불 규정 적용 (위약금 비율 결정)
  → 위약금 > 0원이면:
      1. 보호자에게 위약금 PG 결제 요청
      2. 위약금 결제 완료 → payments에 payment_type='위약금' 행 INSERT
      3. 기존 돌봄비 결제 전액 취소(환불) → status='결제취소'
      4. refunds 행 INSERT (penalty_payment_id = 위약금 결제 건 FK)
  → 위약금 = 0원이면:
      1. 기존 돌봄비 결제 전액 취소(환불) → status='결제취소'
      2. refunds 행 INSERT (penalty_payment_id = NULL)
```

### 2-4. 요청자별 환불 처리 차이

| 요청자 | 위약금 적용 | 비고 |
|--------|-----------|------|
| **보호자** | 시간대별 규정 적용 (0%, 50%, 100%) | 위약금 > 0이면 별도 결제 |
| **유치원 거절** | 위약금 없음 (전액 환불) | 유치원 사정으로 거절 |
| **유치원 취소** | 위약금 없음 (전액 환불) | 유치원 사정으로 취소 |
| **관리자** | 상황에 따라 판단 (위약금 면제 가능) | 관리자 직권. 시간대별 규정이 자동 적용되지 않으며, 위약금 금액을 임의로 지정하거나 면제 가능. 이번 리팩터링에서는 관리자 직권 취소 + 위약금 발생 시나리오의 테스트 데이터는 생성하지 않음 (refund #5는 관리자 강제 취소 → 전액 환불, 위약금 없음으로 유지) |

## 3. 확정된 DB 변경 사항

### 3-1. payments 테이블

| 변경 | 상세 |
|------|------|
| **컬럼 추가** | `payment_type text NOT NULL DEFAULT '돌봄'` |
| **CHECK 제약** | `CHECK (payment_type IN ('돌봄', '위약금'))` — 코드 테이블 없이 CHECK으로 관리 |
| **DEFAULT 처리** | 기존 모든 행은 `'돌봄'`으로 자동 분류 |
| **위약금 행 특성** | `care_fee`, `walk_fee`, `pickup_fee`는 `NULL` (absent, 0이 아님), `amount`에만 위약금 금액 |
| **예약 연결** | `member_id`, `kindergarten_id`, `reservation_id`는 원 예약과 동일 (하나의 예약에 돌봄+위약금 2행 가능) |
| **인덱스** | `idx_payments_payment_type`, `idx_payments_type_status_paid` |

### 3-2. refunds 테이블

| 변경 | 상세 |
|------|------|
| **컬럼 추가** | `penalty_payment_id uuid REFERENCES payments(id)` — nullable |
| **기존 컬럼** | `penalty_tx_id`, `penalty_payment_method`, `penalty_payment_status` 등 기존 penalty 관련 컬럼은 **삭제하지 않음** (모바일 앱 참조 가능성, 전체 작업 완료 후 별도 정리) |
| **인덱스** | `idx_refunds_penalty_payment_id` (partial: WHERE penalty_payment_id IS NOT NULL) |

### 3-3. settlements 테이블

| 변경 | 상세 |
|------|------|
| **값 통일** | `transaction_type`: `'돌봄결제'` → `'돌봄'`으로 변경 |
| **CHECK 제약** | `CHECK (transaction_type IN ('돌봄', '위약금'))` |

## 6. 테스트 데이터 시나리오 요약

### 위약금 발생 시나리오 (1건) — 예약 e0e0e0e0-0004

#### 시나리오 흐름

```
[1] 보호자(박준혁)가 해피독 유치원에 초코(반려견) 돌봄 예약 신청
    → 돌봄비 145,000원 결제 완료 (f0f0f0f0-0004)
    → 예약 상태: '수락대기'

[2] 유치원이 요청 수락
    → 예약 상태: '예약확정'
    → reservation_status_logs: { from: '수락대기', to: '예약확정', changed_by: '유치원' }

[3] 보호자가 등원 48시간 전 취소 요청
    → 위약금 산정 기준: 결제 시점 기준 남은 시간 (예약확정 시점 아님)
    → 규정 2 적용: 50% 위약금
    → 위약금 금액: 145,000 × 50% = 72,500원

[4] 위약금 PG 결제 진행
    → payments INSERT: f0f0f0f0-0012, 72,500원, payment_type='위약금', status='결제완료'

[5] 기존 돌봄비 전액 환불 처리
    → payments UPDATE: f0f0f0f0-0004 status → '결제취소'
    → refunds INSERT: 11111111-0004
      - requested_by: '보호자'
      - cancel_reason: '개인 사정으로 취소합니다'
      - applied_rule: '24~72시간 전 취소 – 위약금 50%'
      - refund_rate: 100 (돌봄비 전액 환불)
      - refund_amount: 145,000원 (= original_amount, 돌봄비 전액)
      - penalty_ratio: 50%
      - penalty_amount: 72,500원
      - penalty_payment_id → f0f0f0f0-0012

[6] 예약 상태 변경
    → 예약 상태: '보호자취소'
    → reservation_status_logs: { from: '예약확정', to: '보호자취소', changed_by: '보호자' }
```
