# 우유펫 관리자 대시보드 — 기술 스택 결정서

## 1. 기존 서비스 현황

### 1-1. 현재 인프라 구성

| 구분 | 기술/서비스 | 비용 | 비고 |
|------|-----------|------|------|
| **모바일 앱** | React Native (JavaScript) | — | 외주 개발자가 구현, 소스코드 소유권은 본인 |
| **기존 DB** | MariaDB | 포함 | 스마일서브 호스팅에 포함 |
| **기존 API** | PHP | 포함 | 스마일서브 호스팅에 포함 |
| **기존 관리자 페이지** | 그누보드(PHP) 기반 | 포함 | → HTML/CSS/JS로 새로 구축 중 (본 프로젝트) |
| **메인 서버** | 스마일서브 | 월 ₩3,000 | DB + API + 관리자 페이지 호스팅 |
| **채팅 서버** | 카페24 | 월 ₩132,000 | 실시간 채팅 전용 서버 |
| **도메인** | wooyoopet.com | 연 ₩18,000 | 스마일서브에서 구매 |
| **현재 월 비용** | — | **월 ₩135,000** | 스마일서브 + 카페24 합산 |

### 1-2. 서비스 상태

- **런칭 전** — 실제 사용자 데이터 없음 (DB에 테스트 데이터만 존재)
- 기존 DB 마이그레이션이 아닌 **새 DB에서 깨끗하게 시작** 가능
- 소스코드 소유권이 본인에게 있어 기존 개발자에게 최신 소스코드 수령 가능

### 1-3. 기존 구조의 문제점

| 문제 | 설명 |
|------|------|
| PHP/MariaDB 유지보수 | Claude AI 코딩 효율이 낮음, 비개발자가 직접 수정하기 어려운 구조 |
| 서버 직접 관리 | 백업·보안·복구·모니터링을 수동으로 해야 함 |
| DB 조회 불편 | 웹에서 데이터를 엑셀처럼 직접 조회·수정 불가 |
| 높은 고정비용 | 런칭 전임에도 월 ₩135,000 고정 지출 (채팅 서버가 대부분) |
| 분산 관리 | 메인 서버(스마일서브) + 채팅 서버(카페24) + 도메인(스마일서브) 각각 관리 |

## 2. 기술 스택 결정사항

### 2-1. 전환 후 기술 스택

| 역할 | 기존 | 전환 후 | 비고 |
|------|------|--------|------|
| **DB** | MariaDB (스마일서브) | **Supabase** (서울 리전, PostgreSQL) | 무료 플랜: 500MB DB |
| **API** | PHP | **Supabase 자동 API + Edge Functions** | 별도 서버 불필요 |
| **인증(로그인)** | PHP 세션 | **Supabase Auth** | 관리자 + 앱 사용자 통합 |
| **파일 저장** | 서버 로컬 | **Supabase Storage** | 무료 플랜: 1GB |
| **채팅** | 카페24 서버 | **Supabase Realtime** | 카페24 서버 대체 |
| **관리자 페이지 호스팅** | 스마일서브 | **Cloudflare Pages** | 완전 무료 |
| **도메인** | 스마일서브 (연 ₩18,000) | **Cloudflare Registrar** | 원가 판매 (연 ~₩14,000) |
| **DNS/보안** | 스마일서브 | **Cloudflare** | DDoS 보호 무료 포함 |

## 3. 기술 스택 결정 이유

### 3-1. Supabase 선택 이유

| 이유 | 설명 |
|------|------|
| 웹 대시보드 | 엑셀처럼 데이터 직접 조회·수정 가능 (비개발자 친화적) |
| 자동 관리 | 백업·보안·스케일링을 Supabase가 자동 처리 |
| 무료 플랜 | 500MB DB + 1GB Storage + 월 50만 Edge Functions 호출 — 초기 충분 |
| AI 코딩 호환 | Claude AI가 Supabase(PostgreSQL + JavaScript) 코드를 잘 생성함 |
| 오픈소스 | 서비스 종료 시 Docker로 셀프호스팅 가능 (벤더 락인 최소화) |
| 데이터 이전 용이 | PostgreSQL 기반이라 표준 SQL 도구로 내보내기/가져오기 가능 |
| 올인원 | DB + API + Auth + Storage + Realtime이 하나의 프로젝트에 통합 |

### 3-2. Edge Functions 선택 이유

| 이유 | 설명 |
|------|------|
| 서버 불필요 | 별도 서버 없이 Supabase 안에서 API 구현 |
| 간편 배포 | CLI 한 줄(`supabase functions deploy`)로 배포 완료 |
| 무료 포함 | 무료 플랜에 월 50만 호출 포함 |
| 선택적 사용 | 단순 CRUD는 Supabase 자동 API(PostgREST)로 처리, 복잡한 로직만 Edge Functions — 전체 API의 약 20%만 직접 작성 |

**Edge Functions가 필요한 경우 (전체의 ~20%)**:
- 결제 검증 (이니시스 서브몰 연동)
- 푸시 알림 발송
- 이니시스 서브몰 등록/해지 API 호출
- 정산 처리 배치 로직
- 복합 비즈니스 로직 (노쇼 자동 판정, 자동 완료 처리 등)

**자동 API로 충분한 경우 (전체의 ~80%)**:
- 회원·유치원·반려동물·예약·결제 등 기본 CRUD
- 목록 조회 + 필터 + 검색 + 페이지네이션
- 상세 조회
- 상태 변경 (단순 UPDATE)

### 3-3. Cloudflare Pages 선택 이유

| 이유 | 설명 |
|------|------|
| 완전 무료 | 정적 사이트 호스팅 무료 (빌드 500회/월, 대역폭 무제한) |
| 도메인 원가 | Cloudflare Registrar는 도메인 등록·갱신을 원가 판매 (마진 0원) |
| 서울 데이터센터 | 한국 사용자에게 빠른 응답 속도 |
| 기업 안정성 | NYSE 상장기업 (시가총액 ~39조원), 서비스 종료 리스크 낮음 |
| DDoS 보호 | 무료 플랜에도 DDoS 방어 포함 |
| 올인원 관리 | 도메인 + DNS + 호스팅 + SSL + 보안을 한 곳에서 관리 |

### 3-4. MariaDB/PHP 탈피 이유

| 이유 | 설명 |
|------|------|
| AI 코딩 효율 | PHP는 Claude AI 코드 생성 품질이 JavaScript/TypeScript 대비 낮음 |
| 서버 관리 부담 | 백업·보안패치·장애복구를 직접 해야 함 |
| DB 접근성 | MariaDB는 phpMyAdmin 등 별도 도구 필요, Supabase는 웹 대시보드 내장 |
| 유지보수 구조 | 비개발자가 AI 도구로 수정·관리하기 어려운 PHP 구조 |

### 4-4. 모바일 앱 통합 구조

현재 모바일 앱은 기존 PHP API + MariaDB에 연결되어 있으나, 궁극적으로 **모든 백엔드를 Supabase로 통일**합니다.

```
[현재]                              [전환 후]
모바일 앱                            모바일 앱
  │                                   │
  ├── PHP API ──→ MariaDB             ├── Supabase 자동 API ──→ PostgreSQL
  ├── 카페24 ──→ 채팅                  ├── Supabase Realtime ──→ 채팅
  └── PHP ──→ 인증                     └── Supabase Auth ──→ 인증

관리자 페이지                         관리자 페이지
  │                                   │
  └── (별도 구조)                      └── 동일한 Supabase ──→ 동일한 PostgreSQL
```

> **핵심**: 관리자 페이지와 모바일 앱이 **동일한 Supabase 프로젝트 + 동일한 DB**를 공유. RLS(Row Level Security)로 관리자/보호자/유치원 권한을 분리.

## 7. 핵심 설계 원칙

### 7-1. 비개발자 + AI 코딩 최적화

| 원칙 | 설명 |
|------|------|
| **프레임워크 최소화** | React/Vue 등 프레임워크 없이 순수 HTML/CSS/JS 유지 — 빌드 과정 불필요, AI가 수정하기 쉬움 |
| **Supabase 자동 API 우선** | 직접 코드를 작성하지 않아도 DB 테이블 기반으로 API가 자동 생성됨 |
| **CDN 방식** | supabase-js를 CDN으로 로드 — npm/번들러 없이 `<script>` 태그 하나로 연결 |
| **파일 단위 분리** | 메뉴별로 JS 파일을 분리하여 한 번에 하나의 파일만 수정 가능하게 구성 |
| **웹 대시보드 활용** | Supabase 웹 대시보드에서 직접 데이터 조회·수정·SQL 실행 가능 |

### 7-2. 보안

| 항목 | 방법 |
|------|------|
| 인증 | Supabase Auth (JWT 기반), 관리자 전용 역할(role) 설정 |
| 권한 | RLS(Row Level Security) — DB 레벨에서 접근 제어 |
| HTTPS | Cloudflare 자동 SSL 인증서 |
| DDoS | Cloudflare 무료 DDoS 보호 |
| API 키 | anon key(공개용) + service_role key(서버 전용) 분리 |
| CORS | Supabase 프로젝트 설정에서 허용 도메인 지정 |

### 7-3. 확장성

| 상황 | 대응 |
|------|------|
| 사용자 증가 | Supabase Pro 플랜 전환 (월 $25) |
| DB 용량 증가 | Pro 플랜 8GB, 필요 시 추가 용량 구매 |
| 트래픽 증가 | Cloudflare CDN이 자동 처리, Supabase는 자동 스케일링 |
| 기능 추가 | Edge Functions 추가, DB 테이블 추가 |
| Supabase 서비스 종료 | 오픈소스이므로 Docker로 셀프호스팅 전환 가능 |
